# SPEC: webhook-service module boundaries

Resolves the architecture review (candidates 1, 2, 3, 4 accepted; candidate 5 recorded as
[ADR-0001](adr/0001-reminder-policy-in-prompt.md)). Implementation target for the sub-agent
build loop.

## File layout

```
webhook-service/
  src/
    verify-signature.js     # HMAC check — transport concern, its own module
    fireflies-client.js      # candidate 1
    notifier.js               # candidate 2
    seen-meetings.js          # candidate 3
    handle-webhook.js         # candidate 4 — transport-agnostic core
    app.js                    # createApp(deps) -> Express app, not listening
  test/
    verify-signature.test.js
    fireflies-client.test.js
    notifier.test.js
    seen-meetings.test.js
    handle-webhook.test.js
    app.test.js                # end-to-end smoke test, real HTTP, fake deps — "one test version"
  index.js                    # entrypoint only: builds real deps, calls createApp, listens
  package.json
  .env.example
```

`app.js` exists so the Express wiring itself is testable without a real network call: `createApp(deps)`
returns an app that can be started on an ephemeral port in a test and driven with real HTTP requests,
while `index.js` is reduced to "build real adapters, call `createApp`, listen" — nothing to unit test
there beyond "does it start."

Test runner: Node's built-in `node:test` + `node:assert/strict`. No new dependency — the
service already only needs `express` and `axios` in production.

## Module interfaces

### `verify-signature.js`
```js
function verifySignature({ secret, signatureHeader, rawBody }) -> boolean
```
Pure function. No network, no env var reads — caller passes the secret in. Replaces the old
`verifySignature(req)` which read `process.env` and `req.headers` directly, making it
untestable without constructing a fake Express `req`.

### `fireflies-client.js`
```js
function createFirefliesClient({ apiKey, retries = 10, delayMs = 30_000, sleep, httpPost }) -> {
  fetchSummary(meetingId) -> Promise<{ overview, action_items } | null>
}
```
`sleep` and `httpPost` are injectable (default to a real `setTimeout`-based sleep and
`axios.post`) so tests can skip real delays and fake Fireflies' responses across attempts.
Owns the GraphQL query string and the auth header — nothing outside this module constructs
either.

### `notifier.js`
```js
function createNotifier({ botToken, chatId, opsChatId, httpPost }) -> {
  notifySummary(summary) -> Promise<void>,
  notifyOpsFailure(meetingId, reason) -> Promise<void>,
}
```
Owns message formatting (plain text, no `parse_mode`) and chat routing. Callers name an event
(`notifySummary`, `notifyOpsFailure`) — never construct a Telegram payload themselves.

### `seen-meetings.js`
```js
function createSeenMeetings() -> {
  has(meetingId) -> boolean,
  markSeen(meetingId) -> void,
}
```
`Set`-backed today. A future persistent adapter (e.g. Redis) implements the same two methods —
`handle-webhook.js` never touches a `Set` directly.

### `handle-webhook.js`
```js
async function handleFirefliesWebhook(
  { eventType, meetingId },
  { firefliesClient, notifier, seenMeetings }
) -> { status: 'ignored' | 'duplicate' | 'processed' | 'failed', meetingId }
```
The transport-agnostic core. Takes an already-authenticated, already-parsed event and three
injected adapters; returns a result describing what happened. No Express, no `req`/`res`, no
`process.env` — fully callable from a test with fake adapters and no HTTP server.

### `index.js`
Express-only: reads env vars, constructs the four real adapters once at startup, verifies the
signature, acks the webhook immediately (200), and calls `handleFirefliesWebhook` with the
parsed body. ~25 lines — no branching logic beyond translating req/res.

## Behavior change from the original skeleton

The original handler acked with different response bodies depending on whether the event was
ignored/duplicate/new (`"Ignored"`, `"Already processed"`, `"Processing"`) before doing any
async work. The new version acks once, generically, as soon as the signature is verified, then
runs `handleFirefliesWebhook` fully asynchronously. Fireflies only requires a fast 2xx — the
specific ack body was never meaningful to it. This centralizes all branching (ignored /
duplicate / processed / failed) inside the one testable function instead of splitting it
across pre-ack and post-ack code.

## Test plan

- `verify-signature.test.js`: valid signature passes, tampered body fails, missing header
  fails, malformed header doesn't throw.
- `fireflies-client.test.js`: returns summary on first success; retries N times with injected
  no-op `sleep` then returns `null` on exhaustion; does not call `sleep` after a successful
  attempt.
- `notifier.test.js`: `notifySummary` posts to `chatId` with the overview/action-items text;
  `notifyOpsFailure` posts to `opsChatId`; neither ever sets `parse_mode`.
- `seen-meetings.test.js`: `has` is false before `markSeen`, true after; independent instances
  don't share state (guards against the old module-level-`Set` leakage across tests).
- `handle-webhook.test.js`: ignores non-`"Transcription completed"` events; returns `duplicate`
  on a second call for the same `meetingId` without calling `firefliesClient` again; calls
  `notifier.notifySummary` on success; calls `notifier.notifyOpsFailure` when
  `fetchSummary` resolves `null` or throws.
