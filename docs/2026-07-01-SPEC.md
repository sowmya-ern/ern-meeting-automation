# SPEC: webhook-service module boundaries

Resolves the architecture review (candidates 1, 2, 3, 4 accepted; candidate 5 recorded as
[ADR-0001](adr/0001-reminder-policy-in-prompt.md)). Implementation target for the sub-agent
build loop.

## File layout

```
webhook-service/
  src/
    verify-signature.js     # HMAC check â€” transport concern, its own module
    fireflies-client.js      # candidate 1
    notifier.js               # candidate 2
    seen-meetings.js          # candidate 3
    handle-webhook.js         # candidate 4 â€” transport-agnostic core
    app.js                    # createApp(deps) -> Express app, not listening
  test/
    verify-signature.test.js
    fireflies-client.test.js
    notifier.test.js
    seen-meetings.test.js
    handle-webhook.test.js
    app.test.js                # end-to-end smoke test, real HTTP, fake deps â€” "one test version"
  index.js                    # entrypoint only: builds real deps, calls createApp, listens
  package.json
  .env.example
```

`app.js` exists so the Express wiring itself is testable without a real network call: `createApp(deps)`
returns an app that can be started on an ephemeral port in a test and driven with real HTTP requests,
while `index.js` is reduced to "build real adapters, call `createApp`, listen" â€” nothing to unit test
there beyond "does it start."

Test runner: Node's built-in `node:test` + `node:assert/strict`. No new dependency â€” the
service already only needs `express` and `axios` in production.

## Module interfaces

### `verify-signature.js`
```js
function verifySignature({ secret, signatureHeader, rawBody }) -> boolean
```
Pure function. No network, no env var reads â€” caller passes the secret in. Replaces the old
`verifySignature(req)` which read `process.env` and `req.headers` directly, making it
untestable without constructing a fake Express `req`.

### `fireflies-client.js`
```js
function createFirefliesClient({ apiKey, retries = 10, delayMs = 30_000, sleep, httpPost }) -> {
  fetchSummary(meetingId) -> Promise<{ overview, action_items } | null>
}
```
`sleep` and `httpPost` are injectable (default to a real `setTimeout`-based sleep and
`axios.post`) so tests can skip real delays and fake Fireflies' responses across attempts.
Owns the GraphQL query string and the auth header â€” nothing outside this module constructs
either.

### `notifier.js`
```js
function createNotifier({ botToken, chatId, opsChatId, httpPost }) -> {
  notifySummary(summary) -> Promise<void>,
  notifyOpsFailure(meetingId, reason) -> Promise<void>,
}
```
Owns message formatting (plain text, no `parse_mode`) and chat routing. Callers name an event
(`notifySummary`, `notifyOpsFailure`) â€” never construct a Telegram payload themselves.

### `seen-meetings.js`
```js
function createSeenMeetings() -> {
  has(meetingId) -> boolean,
  markSeen(meetingId) -> void,
}
```
`Set`-backed today. A future persistent adapter (e.g. Redis) implements the same two methods â€”
`handle-webhook.js` never touches a `Set` directly.

### `handle-webhook.js`
```js
async function handleFirefliesWebhook(
  { eventType, meetingId },
  { firefliesClient, notifier, seenMeetings }
) -> { status: 'ignored' | 'duplicate' | 'processed' | 'failed', meetingId }
```
The transport-agnostic core. Takes an already-authenticated, already-parsed event and three
injected adapters; returns a result describing what happened. No Express, no `req`/`res`, no
`process.env` â€” fully callable from a test with fake adapters and no HTTP server.

### `index.js`
Express-only: reads env vars, constructs the four real adapters once at startup, verifies the
signature, acks the webhook immediately (200), and calls `handleFirefliesWebhook` with the
parsed body. ~25 lines â€” no branching logic beyond translating req/res.

## Behavior change from the original skeleton

The original handler acked with different response bodies depending on whether the event was
ignored/duplicate/new (`"Ignored"`, `"Already processed"`, `"Processing"`) before doing any
async work. The new version acks once, generically, as soon as the signature is verified, then
runs `handleFirefliesWebhook` fully asynchronously. Fireflies only requires a fast 2xx â€” the
specific ack body was never meaningful to it. This centralizes all branching (ignored /
duplicate / processed / failed) inside the one testable function instead of splitting it
across pre-ack and post-ack code.

## Test plan

- `verify-signature.test.js`: valid signature passes, tampered body fails, missing header
  fails, malformed header doesn't throw.
- `fireflies-client.test.js`: returns summary on first success; retries N times with injected
  no-op `sleep` then returns `null` on exhaustion; does not call `sleep` after a successful
  attempt.
- `notifier.test.js`: `notifySummary` posts to `chatId` with the overview/action-items text;
  `notifyOpsFailure` posts to `opsChatId`; neither ever sets `parse_mode`.
- `seen-meetings.test.js`: `has` is false before `markSeen`, true after; independent instances
  don't share state (guards against the old module-level-`Set` leakage across tests).
- `handle-webhook.test.js`: ignores non-`"Transcription completed"` events; returns `duplicate`
  on a second call for the same `meetingId` without calling `firefliesClient` again; calls
  `notifier.notifySummary` on success; calls `notifier.notifyOpsFailure` when
  `fetchSummary` resolves `null` or throws.

## Revision (2026-07-02): per-meeting-series routing

Real credentials arrived with a requirement the original single-`TELEGRAM_CHAT_ID` design
didn't anticipate: different Fireflies meeting series route to different Telegram chats (Bond
daily standup, Bond<>Nebula, ERN Daily Sync, ERN Daily Executive Standup), plus a personal-DM
ops chat for failures. This replaces `TELEGRAM_CHAT_ID` with a routing table.

### `fireflies-client.js` (changed)
GraphQL query now also fetches `title`. Return shape is
`{ title, overview, action_items } | null` instead of the bare summary object â€” `title` is
what `meeting-router.js` matches against.

### `meeting-router.js` (new)
```js
function createMeetingRouter(rules) -> { resolveChatId(meetingTitle) -> chatId | null }
```
`rules` is an ORDERED array of `{ match, chatId }`, checked most-specific-first (e.g.
`'Bond <> Nebula'` must precede the looser `'Bond'`, or a Bond<>Nebula meeting would resolve to
the Bond Team chat instead). Substring match via `.includes()`. Returns `null` when nothing
matches â€” callers must not silently drop that case.

### `notifier.js` (changed)
```js
function createNotifier({ botToken, opsChatId, httpPost }) -> {
  notifySummaryTo(chatId, summary) -> Promise<void>,
  notifyOpsFailure(meetingId, reason) -> Promise<void>,
  notifyUnrouted(meetingId, meetingTitle, summary) -> Promise<void>,
}
```
`chatId` is now a parameter, not constructor config â€” the destination is resolved by
MeetingRouter, not fixed at startup. `notifyUnrouted` is the safety net: a summary that
fetched fine but matched no routing rule goes to `opsChatId` with a note, instead of vanishing.

### `handle-webhook.js` (changed)
Takes a fourth adapter, `meetingRouter`. After a successful `fetchSummary`, resolves
`meetingRouter.resolveChatId(summary.title)`; a `null` result returns `{ status: 'unrouted',
meetingId }` and calls `notifier.notifyUnrouted` instead of `notifySummaryTo`.

### Test additions
- `meeting-router.test.js`: most-specific-rule-wins ordering, fallback to a looser rule,
  distinct resolution for two rules sharing a prefix (`ERN Daily Executive Standup` vs
  `ERN Daily Sync`), `null` on no match, `null` on missing/empty title without throwing.
- `handle-webhook.test.js` / `app.test.js`: added an `unrouted` case alongside the existing
  ignored/duplicate/processed/failed cases.

## Revision (2026-07-02): shared message format between pre-meeting agenda and post-meeting summary

The pre-meeting Cloud Routine's agenda message (prose-only, in
[routines/pre-meeting-reminder.md](../routines/pre-meeting-reminder.md)) was given a concrete
per-person structure (opening line, `@handle` mentions, `<series> Agenda` title, one section per
attendee). This revision brings the webhook service's post-meeting summary to the same visual
style â€” opening line, mentions, `<title> Summary` heading â€” so a team sees a consistent format
whether the message came from the routine or the webhook. Fireflies' `action_items` stays one
unstructured string (Fireflies doesn't return it grouped per attendee), so unlike the agenda
this doesn't split action items per person â€” only the header/mentions match.

### `attendee-handles.js` (new)
```js
function handleFor(displayName) -> string
```
Pure lookup, no injected deps. Maps a Fireflies attendee display name to its Telegram `@handle`
per a small hardcoded table (kept in code, not prose, since `notifier.js` â€” unlike the routine
prompt â€” needs an actual value to build a payload); an unmapped name falls back to the plain
`displayName` rather than being dropped, so an unrecognized attendee still appears in the
message.

### `fireflies-client.js` (changed again)
GraphQL query now also fetches `meeting_attendees { displayName }`. Return shape is
`{ title, attendees, overview, action_items } | null` â€” `attendees` is a `string[]` of display
names, `[]` when Fireflies returns none.

### `notifier.js` (changed again)
```js
function createNotifier({ botToken, opsChatId, httpPost }) -> {
  notifySummaryTo(chatId, summary) -> Promise<void>,
  notifyOpsFailure(meetingId, reason) -> Promise<void>,
  notifyUnrouted(meetingId, meetingTitle, summary) -> Promise<void>,
}
```
Interface unchanged; `summary` now carries `attendees`, consumed by an internal
`formatSummaryBody(summary)` helper shared by `notifySummaryTo` and `notifyUnrouted` so both
paths render the same opening-line/mentions/title layout via `attendee-handles.js`.

### Test additions
- `attendee-handles.test.js`: known display names resolve to their mapped handle; an unmapped
  name falls back to the plain display name.
- `fireflies-client.test.js`: `attendees` is populated from `meeting_attendees` when present;
  existing fixtures updated to expect `attendees: []` when Fireflies returns none.
- `notifier.test.js`: `notifySummaryTo` opens with an `@mention` line built from
  `attendee-handles.js` (including the unmapped-name fallback) and a `<title> Summary` heading.

## Revision (2026-07-03): live summarization + HTML rendering

See [ADR-0003](adr/0003-live-summarization-in-webhook-service.md) for why a live model call was
added at all.

### `summarizer.js` (new)
```js
function createSummarizer({ apiKey, model = 'claude-sonnet-5', maxTokens = 1024, httpPost }) -> {
  simplify(summary) -> Promise<{ overview, action_items }>
}
```
Calls the Anthropic Messages API directly via the injectable `httpPost` (same convention as
`fireflies-client.js`/`notifier.js` â€” no live key needed in tests). Builds a prompt instructing
the model to respond in a strict `OVERVIEW:`/`ACTION_ITEMS:` format and parses it back out via
regex; throws if the API call fails or the response doesn't match that format, so the caller's
fallback logic decides what happens next â€” this module never silently swallows a failure.

### `handle-webhook.js` (changed)
New optional `summarizer` adapter. A `simplifyOrFallback(summarizer, summary)` helper calls it
between `fetchSummary` and routing/notifying; on any thrown error, or when no summarizer is
configured at all, `summary` passes through unchanged. A summarizer failure is never treated as
a processing failure â€” no `notifyOpsFailure`, no `failed` status â€” the meeting still gets a
(raw) summary, preserving "no lost summaries."

### `notifier.js` (changed again)
Switched from no `parse_mode` to `parse_mode: 'HTML'`. New `escapeHtml()` (escapes only `&`,
`<`, `>`) is applied to every interpolated value â€” title, attendee mentions, meetingId, reason,
overview, action items. A `withBoldMarkers()` helper runs `escapeHtml()` first, then converts
`**word**` sequences into `<b>word</b>` â€” the only way bold text reaches Telegram, since raw
HTML from Fireflies or the summarizer is never trusted directly. This replaces the original
"never set parse_mode" decision (still correct in spirit â€” untrusted text must never be
interpreted as markup â€” just re-implemented with an explicit escape step instead of by avoiding
parse_mode entirely, now that bolding is a real requirement).

### `app.js` / `index.js` (changed)
`createApp` takes an optional `summarizer` and threads it into `handleFirefliesWebhook`.
`index.js` constructs `summarizer` only when `ANTHROPIC_API_KEY` is set; otherwise it's
`undefined` and the raw-summary fallback path is always taken.

### Test additions
- `summarizer.test.js`: parses a well-formed `OVERVIEW:`/`ACTION_ITEMS:` response; rejects when
  the API call fails; rejects when the response is missing the expected markers.
- `handle-webhook.test.js`: uses the summarizer's output when one is configured and succeeds;
  falls back to the raw summary (and does **not** call `notifyOpsFailure`) when it throws.
- `notifier.test.js`: asserts `parse_mode === 'HTML'` (all four send paths); asserts HTML
  special characters are escaped and `**bold**` markers become `<b>` tags in the same message.

## Revision (2026-07-03, part 2): architecture review + pre-meeting relay endpoint

Architecture review (`/improve-codebase-architecture`) found 3 candidates; 2 applied in code,
1 correctly rejected as ADR-blocked (see CLAUDE.md's status log for the full breakdown).
Applied: `bold-marker.js` (new) â€” the one shared contract `summarizer.js` and `notifier.js`
both depend on, plus `summarizer-notifier-integration.test.js` proving they actually agree
end-to-end. `routing-table.js` (new) â€” extracted `index.js`'s previously inline, untested
4-rule table, with `assertOrderingIsSafe(rules)` called at startup so a misordered rule fails
fast instead of silently misrouting.

Separately, see [ADR-0004](adr/0004-pre-meeting-relay-instead-of-routine-secrets.md) for a new
relay endpoint resolving the pre-meeting Cloud Routine's secret-handling problem.

### `verify-relay-token.js` (new)
```js
function verifyRelayToken({ secret, authHeader }) -> boolean
```
Timing-safe `Authorization: Bearer <token>` check, same shape as `verify-signature.js`.

### `relay-chat-keys.js` (new)
```js
function buildRelayChatMap(env) -> Record<string, chatId>
function resolveRelayChatId(map, chatKey) -> chatId | null
```
Maps symbolic keys (`BOND_TEAM`, `BOND_NEBULA`, `ERN_EXEC_STANDUP`, `ERN_SUPER_TEAM`, `OPS`) to
real chat IDs â€” the pre-meeting routine only ever sends the symbolic key.

### `notifier.js` (changed again)
New `sendPlainText(chatId, text)` â€” no `parse_mode`, sends the routine's already-composed text
verbatim. `send()`'s `parseMode` parameter is now overridable (defaults to `'HTML'`) rather than
hardcoded, so this path can opt out.

### `app.js` / `index.js` (changed again)
New route `POST /relay/telegram-agenda`: checks `verifyRelayToken`, resolves `chatKey` via
`relayChatMap`, calls `notifier.sendPlainText`. 401 on a bad/missing token, 400 on an unknown
`chatKey` or missing `text`. `index.js` constructs `relayChatMap` from `process.env` and passes
`relaySecret`/`relayChatMap` into `createApp`.

### Test additions
- `verify-relay-token.test.js`, `relay-chat-keys.test.js`: unit tests for the two new pure
  modules.
- `notifier.test.js`: `sendPlainText` sends verbatim text with no `parse_mode`.
- `app.test.js`: 3 new smoke tests â€” valid relay request sends to the resolved chat; wrong
  token â†’ 401, never sends; unknown `chatKey` â†’ 400, never sends.
